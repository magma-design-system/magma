The `<mds-keyboard>` web component renders a keyboard shortcut as a row of physical-looking keycaps, acting as the compound parent for one or more slotted `<mds-keyboard-key>` children. It has no direct HTML equivalent; it visualizes a key combination (e.g. `Ctrl + C`) and can optionally let the user verify that combination by typing it.

#### Semantic Behavior

- **Default slot is the key sequence**: the component projects slotted `<mds-keyboard-key>` elements into a `.shortcuts` container and automatically inserts a `+` separator (a small `mds-text`) between consecutive keys.
- **Compound parent**: it reads each child key's `name` to build the expected key-code combination; the children supply the visual keycaps and, during a test, are toggled to their `pressed` state by the parent.
- **Test mode (`try`)**: when enabled the component appends a trigger button and a tooltip; clicking the button captures `keydown`/`keyup` on the focused shortcut area and compares the typed combination against the slotted keys.
- **Test lifecycle and focus**: starting a test moves focus and `tabindex` onto the shortcut area and puts the trigger button into its `await` state; the result is committed on `keyup`, after which focus and pressed states are reset, and blurring the area aborts the test.
- **Result feedback**: a pass shows a "done" icon, a fail shows a "close" icon, and the tooltip surfaces the localized success or error message; the outcome is also reflected on the host via the `test` attribute.
- **Localization**: trigger title, tooltip copy, and error messages resolve through the shared locale system (`el`/`en`/`es`/`it`), and `updateLang()` can re-resolve the active language at runtime.

#### Properties & Visual Configurations

This component does not use the shared `variant` / `tone` ladders from [`projects/stencil/SPEC.md`](../../../../SPEC.md#tone-and-variant-system); those props on the internal trigger button are fixed implementation details, not public configuration.

#### Other behavioral props

- **`try`** opts the keyboard into interactive verification mode. Enable it only when you want the user to be able to confirm a shortcut by typing it; leave it unset for a purely presentational shortcut display.
- **`test`** reflects the verification outcome (`'pass'` / `'fail'`) onto the host. It is primarily driven internally by the test flow and is mainly useful as a styling/state hook rather than a value you set yourself.
