The `<mds-card-footer>` web component is the footer region of a [`<mds-card>`](../../mds-card), grouping the trailing content of a card (typically actions such as buttons or supplementary metadata) into the card's bottom slot.

#### Semantic Behavior

- **Compound child only**: `<mds-card-footer>` must be a direct slot child of `<mds-card>`; it is never used standalone, nor mixed in as a different child type. The parent collects it through its named `footer` slot.
- **Self-slotting**: The host renders with `slot="footer"`, so simply placing it inside `<mds-card>` automatically routes it into the parent's footer region without the author setting the slot manually.
- **Pure layout wrapper**: It is a shadow-DOM container with a single default slot and no props, no state, no events, and no ARIA role of its own. It contributes only structural placement, not semantics.
- **Parent-driven responsiveness**: Its position and responsive reflow are governed by the parent card's layout (the card's container-query / `auto-grid` behavior); the footer itself does not manage spacing or breakpoints.
- **Presence affects parent layout**: `<mds-card>` detects which `mds-card-*` children are present to compute its layout, so including or omitting the footer changes how the card arranges its regions.

#### Slot semantics

This component exposes a single default slot and has no configurable properties. Pass any `text`, `HTML elements`, or other components (commonly `<mds-button>` actions) into the default slot; they will be rendered in the card's footer region. Because layout, alignment, and responsive behavior are owned by the parent `<mds-card>`, the footer's only job is to mark its children as the card's footer content.
