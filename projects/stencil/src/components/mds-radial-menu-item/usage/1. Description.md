The `<mds-radial-menu-item>` web component is a single actionable spoke of the [`<mds-radial-menu>`](../../mds-radial-menu) compound. It is not a primitive of its own: it wraps an `<mds-button>` (rendered as a circular icon button) plus an optional `<mds-tooltip>`, and is positioned along the menu arc by its parent.

#### Semantic Behavior

- **Compound child only**: Must be a direct slot child of `<mds-radial-menu>` — it renders with `<Host slot="item">` so the parent can collect it via `slot="item"`. It is never used standalone, and the radial menu's `item` slot should contain only `<mds-radial-menu-item>` elements, not other child types.
- **Parent-driven sizing and ordering**: The parent overrides each item's `size` to match the menu and assigns its angular position by setting the private `--mds-radial-menu-item-index` CSS variable; an item's `size` prop is therefore effectively a fallback that the menu will normalize on load and on size changes.
- **Open/close animation**: Items are hidden (`transform: scale(0)`, `pointer-events: none`) until the parent menu is opened, then translate outward along the arc using the inherited transition timing — the item itself carries no `opened` state.
- **Conditional tooltip**: An `<mds-tooltip>` is rendered (anchored to the inner button, placed `top`) only when `tooltip` is set; otherwise no tooltip node exists.
- **Accessibility**: The `tooltip` string is also passed through as the inner button's `aria-label`, so it doubles as the accessible name of the icon-only action.
- **No own events**: The item emits no custom events; activation is the inner `<mds-button>`'s native click, which the consumer wires up directly on the element.

#### Properties & Visual Configurations

The shared `variant` / `tone` / `size` ladders are defined in [`projects/stencil/SPEC.md`](../../../../SPEC.md#tone-and-variant-system); they are forwarded verbatim to the inner `<mds-button>`. Defaults are deliberately distinct from a standalone button: `variant` defaults to `dark` and `tone` to `weak` so items read as secondary spokes against the parent's trigger button (which defaults to the `strong` tone). Set `icon` to the glyph that represents the action — it is the primary content of each item since the label lives only in the `tooltip`. Use `tooltip` both to describe the action on hover and to supply the accessible name; omit it only when the icon is unambiguous and labelled elsewhere. Prefer leaving `size` unset and letting the parent menu drive it for visual consistency across all spokes.
