Abbiamo regole molto precise su come possono essere formattati i nostri messaggi git commit. Questo porta a messaggi più leggibili che sono facili da seguire quando si guarda attraverso la cronologia del progetto. Ma usiamo anche i messaggi git commit per generare il log delle modifiche del Design System.

<Image aspectRatio="2:1" aspectRatioPosition="center 65%" src={require('./coding-bro.svg')} />

## Formato messaggio delle commit

Ogni messaggio di commit è costituito da un **header**, un **body** e un **footer**. L'header ha un formato speciale che include un **type**, uno **scope** e un **subject**:

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

L'**header** è obbligatoria e lo **scope** è facoltativo.

```
docs(design-system): update design-system to beta.5
```

Ogni riga del messaggio di commit non deve essere più lunga di 100 caratteri. Ciò consente di leggere più facilmente il messaggio su [GitLab][gitlab] e in vari strumenti git.

Il **footer** dovrebbe contenere un [riferimento di chiusura][github-docs] a un problema, se presente.

```
fix(design-system): need to depend on latest rxjs and zone.js

The version in our package.json gets copied to the one we publish, and users need the latest of these.
```

#### Revert

Se il commit fa un **revert** a un commit precedente, dovrebbe iniziare con `revert:`, seguito **dall'header** del commit ripristinato.

Nel corpo dovrebbe dire: `Questo ripristina il commit <hash>.`, Dove l'hash è **l'SHA** del commit che viene ripristinato.

## Type

Deve essere uno dei seguenti:

- **build**: Modifiche che influenzano il sistema di build o le dipendenze esterne (ambiti di esempio: gulp, broccoli, npm)
- **change**: Modifica di una funzionalità che cambia la API di funzionamento
- **ci**: Modifiche ai nostri file e script di configurazione CI (ambiti di esempio: Travis, Circle, BrowserStack, SauceLabs)
- **docs**: Modifiche alla documentazione
- **feat**: Aggiunta di una nuova feature
- **fix**: Un bug fix
- **perf**: Modifica al codice che migliora le performance
- **refactor**: Modifica al codice che non fixa bug o aggiunge delle feature
- **style**: Modifiche che non influiscono sul codice (stile, spazi, formattazione, punti e virgola mancanti, ecc.), **non si applica allo scope** `styles`
- **test**: Aggiunta di test mancanti o correzione di test esistenti

## Scope

Lo **scope** essere il nome del progetto **npm** interessato visto dalla persona che legge il log delle modifiche generato dai messaggi di commit.

Di seguito è riportato l'elenco degli scope supportati:

- **angular**
- **font-icons-cli**
- **mgg-icons**
- **react**
- **styles**
- **tokens**
- **web-component**

---

#### Subject

Il **subject** contiene una breve descrizione della modifica:

- Usa l'imperativo, il tempo presente: "cambia" (change) non "cambiato" (changed ne changes)
- Non scrivere in maiuscolo la prima lettera
- Nessun punto (.) alla fine

#### Body

Proprio come per il **subject**, usa l'imperativo, il tempo presente: "cambiamento" non "cambiato" né "cambiamenti". Il **body** dovrebbe includere la motivazione per il cambiamento e contrastarla con il comportamento precedente.

#### Footer

Il **footer** dovrebbe contenere tutte le informazioni sulle **Breaking Changes** ed è anche il luogo in cui fare riferimento ai problemi di [GitLab][gitlab] che questo commit **chiude**.

Le **Breaking Change** dovrebbero iniziare con la parola **BREAKING CHANGE:** con uno spazio o due nuove righe. Il resto del messaggio di commit viene quindi utilizzato per questo.
